## 2. 用户本轮真实测试结果

测试环境：Windows 11、真实麦克风、真实百炼转录；分别测试 360 开启和关闭。

| 场景 | 实际结果 | 结论 |
|---|---|---|
| 记事本 | 能自动写入 | 原生 `Edit/RichEdit + EM_REPLACESEL` 路径有效 |
| 微信输入框 | 胶囊显示“已保存”，没有自动写入 | 开始录音时被判定为没有可编辑目标，根本没有进入自动写入路径 |
| Codex 输入框 | 胶囊显示“未能写入”，没有自动写入 | 已进入剪贴板/键盘粘贴路径，但 `Ctrl+V` 没有被目标接受 |
| 历史记录 | 微信和 Codex 的识别文字都能找到并复制 | 转录、持久化和复制兜底正常 |
| 360 开启/关闭 | 微信和 Codex 均不能写入 | 本轮主要问题不是 360；不要继续把失败归因于安全软件 |
| 胶囊 | 整体偏大；声波变化太小；静音仍显示高低不一的柱 | 尺寸和音量可视化不符合用户预期 |

“已保存”和“未能写入”的当前含义不同：

- “已保存”：`CapturedInputTarget.Kind` 不是 `Editable`，没有尝试写入，只保存历史。
- “未能写入”：识别到了候选目标，尝试粘贴但没有获得成功结果，文字留在剪贴板或历史。

用户无法从这两个短文案直接理解差异，v0.6.2 必须调整状态语义，见第 6 节。

## 3. 当前代码诊断

### 3.1 胶囊为什么显得大

`OverlayWindow.xaml` 当前使用 `204 × 58 DIP`、左右 `22 DIP` 内边距、17 号粗体、76 DIP 声波列。在 125%/150% DPI 下按物理像素放大后，视觉面积明显偏大。

### 3.2 声波为什么变化太小、静音也不是横线

`OverlayWindow.SetAudioLevel` 当前算法：

```csharp
smoothedLevel = smoothedLevel * 0.72 + level * 0.28;
var scale = 0.18 + smoothedLevel;
```

问题：

1. 固定加上 `0.18`，即使输入为 0，五根柱也始终保持不同高度，不可能显示静音横线。
2. 录音层传入的是单个缓冲区的最大峰值；峰值容易受底噪和瞬态影响，不适合表现“正在说话”。
3. 平滑系数压缩了变化范围，又没有噪声门、分贝映射、迟滞或静音状态。

### 3.3 微信为什么显示“已保存”

`CaptureCurrentTarget()` 只通过 Win32 `GetGUIThreadInfo`、窗口类名和原生文本容量判断可编辑性。当前类名规则仅覆盖：

- `Edit/RichEdit`
- 名称含 `Chrome`
- 名称含 `WebView`
- `Internet Explorer_Server`
- `HwndWrapper`

新版微信的输入区可能没有独立 Win32 `Edit` 子窗口，或焦点句柄/类名不在上述规则内，因此被判为 `InputTargetKind.None`。不能只简单把整个微信进程永久加入可编辑白名单，否则光标不在输入框时也可能误触发粘贴快捷键。

### 3.4 Codex 为什么进入粘贴路径仍失败

Codex 属于 Chromium/Electron/WebView 类目标。当前流程把文字写入剪贴板后调用一次 `SendInput(Ctrl+V)`，并保留剪贴板。`SendInput` 返回 4 只能证明四个键盘事件已进入系统输入队列，不能证明 Codex 编辑器收到或执行了粘贴。

当前缺少足够诊断数据，不能在没有证据的情况下继续猜测。需要先记录安全的目标元数据，再验证以下可能性：

- 捕获到的是顶层 Chromium 窗口而不是实际编辑区域。
- 录音期间焦点句柄变化，结束时严格的句柄相等校验产生错误目标或 Lost。
- 粘贴发生前仍存在 Alt/Ctrl/Win 修饰键状态。
- 目标窗口虽然仍在前台，但实际 UI Automation 焦点不再位于可编辑控件。
- Codex/微信对注入键盘事件的处理方式与原生窗口不同。

## 4. P0：先增加安全诊断，不允许盲修

修改 `CapturedInputTarget` 或增加独立 `InputTargetDiagnostics`，在设置页“最近目标”区域或 Debug 日志记录以下信息：

- 捕获阶段：`ProcessName`、进程 ID、前台窗口类名、焦点窗口类名、顶层 HWND、焦点 HWND、`hwndCaret`、`InputTargetKind`。
- UI Automation：聚焦元素的 `ControlType`、`ClassName`、`AutomationId`、`IsKeyboardFocusable`、`IsEnabled`、是否支持 `ValuePattern/TextPattern`；不要记录控件文字。
- 插入前复核：当前前台 HWND/进程是否与开始录音时相同，当前 UIA 焦点是否仍可编辑。
- 键盘状态：`VK_RMENU/VK_MENU/VK_CONTROL/VK_LCONTROL/VK_RCONTROL/VK_LWIN/VK_RWIN` 是否已释放。
- 写入结果：选择的策略、`SendInput` 返回数量、`Marshal.GetLastWin32Error()`、是否保留剪贴板。

严禁记录：转写全文、旧剪贴板内容、API Key、录音内容或用户聊天内容。发布版默认不写详细日志；可通过设置页临时打开“兼容性诊断”，并提供清除按钮。

先用真实微信和 Codex 收集一次上述元数据，再决定最终分类规则。不要仅靠进程名硬编码。

## 5. P0：重构输入目标捕获与自动写入

### 5.1 增加 UI Automation 目标检测

在现有 Win32 检测之外增加 UI Automation：

1. 录音开始时同时取得 `AutomationElement.FocusedElement`。
2. 元素满足以下任一条件时标记为可编辑候选：
   - `ControlType.Edit` 且可获得键盘焦点；
   - 支持可写 `ValuePattern`；
   - 支持文本模式且其父级/祖先明确为可编辑输入控件。
3. 保存顶层窗口、进程和 UIA RuntimeId/必要元数据，不保存输入框内容。
4. 结束识别后重新取得当前 UIA 焦点。只要顶层窗口和进程仍与开始时一致、当前焦点仍是可编辑候选，就允许更新候选元素；不要强制要求整个录音期间焦点 HWND 永远完全相同。
5. 真正无输入框时仍保持 `None`，不能向微信主窗口或 Codex 页面任意发送粘贴。

不要直接用 `ValuePattern.SetValue(finalText)` 覆盖输入框现有内容。除非能保持选区和光标语义，否则 UIA 只用于判定目标，实际插入仍优先采用剪贴板粘贴。

### 5.2 粘贴前的安全条件

执行键盘粘贴前必须再次检查：

1. 目标顶层窗口仍是当前前台窗口。
2. 目标进程 ID 未变化。
3. UIA 当前焦点仍为可编辑候选。
4. 右 Alt、Alt、Ctrl、Win 均已释放；最多等待 300 ms，每 15～20 ms 检查一次。不要伪造松开用户正在真实按住的键。
5. 胶囊窗口仍保持 `NoActivate`，不得调用会抢走输入焦点的 `Activate()`。

### 5.3 写入策略

建议顺序：

1. 原生 `Edit/RichEdit`：继续使用 `EM_REPLACESEL` 并校验结果。
2. UIA 确认的 Chromium/Electron/WebView/微信候选：将最终文字放入剪贴板，等待剪贴板可读回，再发送一次 `Ctrl+V`。
3. 发送后不自动发送 Enter，不自动连续尝试第二种粘贴快捷键，因为无法验证时重复尝试可能造成双份文字。
4. 未确认成功时保留最终文字在剪贴板，状态显示“已复制”，历史继续保存。
5. 目标消失或焦点离开输入框时不粘贴到新窗口。

如果真实诊断证明 Codex 或微信不接受当前 `SendInput`，可在独立实验分支比较以下方案，但必须每种方案单独人工测试，不能全部连续执行：

- 使用扫描码形式的 `SendInput` 发送 Ctrl+V。
- 对明确支持的应用选择 `Shift+Insert`，但不得与 Ctrl+V 连续双发。
- 使用 UI Automation 将键盘焦点恢复到原候选元素后再粘贴；只有顶层窗口始终为当前前台窗口时允许。

禁止使用 DLL 注入、驱动、剪贴板监听劫持、关闭 360、关闭 UAC 或默认管理员常驻。

## 6. P0：统一结果状态和文案

v0.6.2 胶囊状态必须改为用户能直接理解的结果：

| 条件 | 胶囊标签 | Toast |
|---|---|---|
| 原生路径确认写入 | `已写入` | `识别结果已写入并保存` |

不要再用单独的“未能写入”作为胶囊最终标签，因为它没有告诉用户文字已经安全复制。设置页可以保留更详细的策略和失败原因。

## 7. P1：胶囊尺寸和声波视觉

### 7.1 胶囊新尺寸

将设计规格从 `204 × 58 DIP` 调整为：

- Window/Border：`184 × 50 DIP`
- 圆角：`25`
- 水平内边距：`18`
- 主文字：`16`、Bold
- 分隔线：`1 × 20`
- 右侧声波列：`64`
- 阴影：BlurRadius `26`、ShadowDepth `6`，保持现有明亮玻璃配色
- 与任务栏距离仍为 `12 DIP`

所有尺寸仍使用 DIP，并保持 Per-Monitor V2。必须在 100%、125%、150% 缩放下截图比较，不要只看单一显示比例。

### 7.2 静音必须是一条横线

录音态的右侧区域拆成两个互斥视觉：

```text
静音：        ─────────
说话：        ▁ ▄ █ ▅ ▂
```

- 新增 `SilenceLine`：宽 28、高 2、全圆角、蓝紫渐变或 PrimaryBrush。
- 新增/保留 `WaveBarsPanel`：5 根柱，宽 3，间距 2，最大高度约 28。
- 静音时只显示 `SilenceLine`，完全隐藏柱状声波。
- 检测到说话时隐藏横线，显示明显起伏的五根柱。

### 7.3 音量算法

不要继续用单缓冲区绝对峰值直接驱动 UI。建议计算 RMS 并映射为 dBFS：

```text
rms = sqrt(sum(sample²) / sampleCount) / 32768
db = 20 * log10(max(rms, 0.000001))
visual = clamp((db + 50) / 35, 0, 1)
```

推荐初始参数，允许根据真实麦克风调整：

- 进入说话态：高于 `-42 dBFS`
- 退出说话态：低于 `-48 dBFS` 持续 180～250 ms
- Attack：0.55～0.70，保证开口立即跳动
- Release：0.15～0.25，防止声波突然归零
- 非线性放大：`Pow(visual, 0.55～0.70)`，让普通说话也有明显高度
- 五柱高度权重可用 `[0.55, 0.85, 1.0, 0.75, 0.50]`

需要迟滞和静音保持时间，避免底噪造成横线与声波快速闪烁。声波只能由真实音量驱动，不做与声音无关的循环动画。

## 8. 自动测试要求

至少新增以下测试：

1. PCM 全零输入映射为静音状态。
2. `short.MinValue` 不溢出。
3. 低于退出阈值持续规定时间后显示横线。
4. 高于进入阈值时声波高度明显大于静音基线。
5. UIA 可编辑候选和非编辑元素分类测试。
6. 开始时 `None`、结束时同一前台应用出现可编辑 UIA 焦点时可更新目标。
7. 顶层窗口或进程变化时必须判定 Lost，不得写入。
8. 修饰键未释放时等待/放弃，不发送 Ctrl+V。
9. `SendInput` 返回 4 仍不得标记为“已确认写入”。
10. `InsertionBlocked/CopyFallback` 映射为“已复制”，`Saved` 只用于未尝试写入。
